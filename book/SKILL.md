# 《대치동 윤원장의 북스펙》 집필 시스템

## 핵심 아이디어
"문제와 답을 이어 놓으면 그것이 곧 책이 된다." 편집장 인터뷰 질문 40~50개에 저자 목소리로 답하고, 그 문답을 이어 한 권을 만든다. 매일 새 자료가 들어오면 페르소나를 갱신하고 관련 문답을 다시 써 완성도를 높인다.

## 파일 구조 (book/)
- `persona-bible.md` — ⭐ 최상위 기준. 정체성·목소리·시그니처 개념·독자·**법적 가드레일**·출력 형식. 집필 전 항상 최신화.
- `question-list.md` — 브랜드 제거된 문항 40개(+확장).
- `source-appendix.md` — 사례·방법·문체 원재료(브랜드 제거).
- `저자-기획안-완성본.md` — 출간 기획서(상업 포지셔닝).
- `draft/part-1..8.md` — 섹션별 문답 초안(직원 산출물).
- `북스펙-초고-v{N}.md` — 조립된 통합 초고.

## ⚠️ 법적 가드레일 (위반 시 전량 폐기)
브랜드명(책나무/BookTree), 프랜차이즈 특허·고유 시스템명, 앱 '북스펙'의 기능 설명을 **절대** 넣지 않는다. → "제가 대치동에서 운영해온 독서논술 학원 / 제 교육 현장 / 전국의 원장님들"로 일반화. 방법론(코칭·북토킹·발문·과정중심)은 일반 개념이라 사용 가능하되 브랜드에 귀속시키지 않는다.

## 워크플로우

### 1) 자료 흡수 → 페르소나 갱신
새 자료를 받으면 사실/사례/방법/문체를 추출해 `persona-bible.md`(정체성·스토리 뱅크·메서드 뱅크)와 `source-appendix.md`에 반영. 브랜드는 그 자리에서 제거.

### 2) 직원(서브에이전트) 병렬 투입
문항을 섹션으로 나눠(기본: 5문항×8섹션) 서브에이전트에게 각 섹션을 배정. 각 에이전트는 persona-bible·question-list·source-appendix를 **정독한 뒤** 저자 1인칭으로 집필해 `draft/part-N.md`에 저장. 교육 파트는 목소리 2-A(따뜻·문학적+실전 행동), 운영 파트는 2-B(구조적+프레임·수치).

### 3) 조립 → 검수
`scripts/assemble.py`로 part-*.md를 순서대로 이어 `북스펙-초고-v{N}.md` 생성. 그다음 **가드레일 검수**(브랜드 잔재 grep) + **목소리 검수**(교차 반복·클리셰·톤 이탈)를 돌린다.

### 4) 사용자 리뷰 → 반복
사용자에게 초고를 보여주고 문항별 피드백을 받아 해당 파트만 재집필. 만족·피드백 소진·진전 정체까지 반복.

### 5) 산출 (선택)
편집장 회의용으로 `docx` 스킬을 사용해 `.docx`로 패키징해 전달.

## 목소리 요약 (전문은 persona-bible 2장)
- 시작은 짧고 감각적인 장면 → 통찰 → 오늘의 행동/원리로 닫기.
- 저자 은유(조소·흙·기다림·우주) 사용, 클리셰·전부 대문자·훈계조 금지.
- 답변 1개당 350~600자 이상. 이으면 한 장(章)이 되게.

## 검수 체크리스트
- [ ] 브랜드/특허/앱명 잔재 0건 (grep: 책나무|BookTree|북트리|BToS)
- [ ] 각 답 = 장면/사례 → 통찰 → 행동·원리
- [ ] 교육 파트에 실전 행동, 운영 파트에 프레임/스크립트/수치
- [ ] 문항 간 사례·표현 중복 과다하지 않음
- [ ] 저자 1인칭·목소리 일관
